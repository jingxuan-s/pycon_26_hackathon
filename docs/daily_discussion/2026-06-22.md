# Daily Discussion Summary - 2026-06-22

## Purpose
Continue refining the SkillsFuture career pathway MVP for hackathon judging. The day focused on Telegram UX polish, explanation quality, scoring methodology clarity, documentation coherence, and privacy-safe report handling.

## Main Decisions
- Keep the current deterministic target-aware suitability score as the main judgment method because it explains readiness and gaps clearly.
- Use vector-space nearest-neighbour discovery only for Explore pathways, then re-rank candidates using the existing suitability and gap-cost logic.
- Use weighted L1 nearest-neighbour discovery for Explore pathways, with a shared-skill overlap guard so unrelated low-requirement roles do not dominate results.
- Show up to six Explore/Advance suggestions in Telegram after the hybrid Explore fix produced relevant data-related roles again.
- Keep related skills as explanation-only. Similar SkillsFuture skills, such as Data Analytics and Computational Modelling versus Computational Modelling, should be shown as related evidence but must not change the score.
- Normal Telegram report and action-plan attachments should be transient and not written to `data/processed`; debug reports remain persisted only when explicitly enabled.

## Dataset / Technical Findings
- The actual skill-vector space has 2,316 unique skill dimensions; 3D diagrams are illustrative only.
- Some SkillsFuture skills overlap semantically but are distinct normalized skill IDs, so silent merging would weaken deterministic traceability.
- Current dataset supports K&A-backed learning actions, not official course recommendations.
- Skill weights remain configurable in methodology but all MVP weights stay `1.0` until a defensible weighting method is introduced.

## Product Direction
- Telegram remains the MVP demo surface, but normal chat output should stay compact and readable.
- The user workflow is explicitly designed around two starting points: upload a resume or choose a current dataset role.
- The goal menu supports four different career questions: Explore pathways, Advance roles, Search target role, and Paste JD.
- Skill review remains mandatory before scoring so users can edit, remove, or add skills before being judged.
- The README should explain this workflow design choice as a user-flexibility decision that supports the hackathon goal.

## Architecture Decisions
- Added in-memory attachment support to `BotResponse` so normal Telegram reports/action plans can be sent without writing local Markdown files.
- Split report rendering from report writing with `render_result_report_markdown(...)`; debug/local workflows can still call `write_result_report(...)`.
- Updated the live Telegram adapter to send `attachment_content` via `BytesIO` when no file path is provided.
- Debug mode still persists debug reports for auditability and validation.
- `.gitignore` now ignores timestamped resume/questionnaire/prototype reports and generated Telegram action-plan artifacts.

## Explainability Decisions
- Related-skill notes appear in score explanations, gaps, reports, and action plans, but they explicitly say why the skill is still a gap under exact SkillsFuture scoring.
- Normal Telegram outputs hide confidence, evidence snippets, mapping type, source rows, parser status, formula details, and debug metadata.
- Debug reports keep parser diagnostics, evidence snippets, mapping type, source rows, formula details, and parser/explainer status.
- Action plans should remain K&A-backed and readable, with clear sections for target proficiency, K&A focus, action, and evidence to build.
