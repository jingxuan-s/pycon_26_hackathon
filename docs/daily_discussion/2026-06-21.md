# Daily Discussion Summary - 2026-06-21

## Purpose
Continue turning the SkillsFuture jobs-skills datasets into an explainable career pathway MVP, with Telegram as the demo surface and Python as the core implementation language. The day focused on shifting from questionnaire-first assumptions to a more robust resume/role-first workflow, improving parser explainability, reducing Telegram clutter, and preserving process evidence for judging.

## Main Decisions
- Treat resume upload and direct role search as first-class entry paths for the MVP.
- Keep the old questionnaire as a legacy/internal path rather than the default `/start` flow.
- Require users to review parsed skills before scoring: edit levels, remove incorrect skills, and add missing SkillsFuture skills.
- Keep deterministic scoring authoritative. Parser and explainer agents may extract, verify, and summarize evidence, but must not calculate suitability or rerank recommendations.
- Use compact Telegram messages by default and reserve detailed evidence, confidence, mapping type, formulas, and source rows for generated reports.
- Hide debug report controls in normal Telegram launches and expose them only with `--debug-mode`.

## Dataset / Technical Findings
- The SkillsFuture datasets provide roles, skill requirements, TSC/CCS codes, proficiency descriptions, and K&A items.
- The current local dataset does not include SkillsFuture course listings or provider/course URLs, so action plans should be framed as learning actions rather than course recommendations.
- K&A rows can support more actionable guidance by showing target proficiency, K&A focus, practical practice tasks, and evidence users should build.
- Parser evidence needs hygiene: contact details, resume headers, and weak education/contact snippets should not become user-visible reasons.

## Product Direction
- Telegram remains useful as the constrained MVP interface, but local workflow refinement remains important for rapid UX iteration.
- The main flow is now: `/start` -> upload resume or search role -> review skill profile -> choose target mode -> score -> explain/gaps/action/report.
- Target modes retained: explore pathways, advance roles, search a specific dataset role, or paste a JD for suitability scoring.
- Recommendation lists should not overload users with gap counts before they choose a target. Gaps are shown after selection.
- Navigation needs back buttons across review, edit/add skill, target selection, recommendations, JD prompt, and result screens.

## Architecture Decisions
- Telegram is kept as a thin adapter over `TelegramCareerBotService` and the existing deterministic resume recommender/scoring modules.
- Added active-score routing so `Why this score`, `Show gaps`, and `Generate action plan` work for resume/JD results instead of requiring the old questionnaire pathway state.
- Added normal/debug report modes:
  - Normal report: confirmed skills, suitability, gaps, and learning actions.
  - Debug report: parser diagnostics, evidence snippets, mapping type, confidence, source rows, formula, and status.
- Added `--debug-mode` to the live Telegram adapter to expose debug-report buttons only during debugging.

## Explainability Decisions
- Normal score explanation should say what skills matched, what the biggest gaps are, and why the percentage landed where it did.
- Skill parser output shown to users should use plain-language reasons instead of raw evidence and mapping metadata.
- Action plans should include TSC/CCS code, target level, target proficiency, K&A focus, practical action, and evidence to build.
- Debug/audit reports still preserve the traceability needed for judging and developer review.
